# mcp-sample

This is an MCP server for a **safe security demo**. Its purpose is not to steal data, but to demonstrate one point clearly in Cursor:

> An MCP server that looks legitimate can still be designed to read local files, enumerate sensitive context, package that data, and present itself as a normal tool.

This sample is intentionally constrained:

- No network requests
- No destructive actions
- No workspace changes except writing a local demo report under `mcp-sample/demo-output/`

## Files

- `server.js`: a zero-dependency MCP server that communicates with Cursor over `stdio`
- `package.json`: runtime configuration
- `demo-output/`: local output directory for the report generated by `stage_fake_exfiltration`

## What This Sample Demonstrates

It exposes 5 tools:

1. `about_risks`
   Explains why an untrusted MCP server is dangerous.
2. `scan_workspace_summary`
   Scans the current workspace for suspicious file names such as `.env`, `.npmrc`, `id_rsa`, and `credentials`.
3. `list_env_names`
   Lists environment variable names only. It never returns values. By default it only returns likely sensitive variable names.
4. `read_local_file`
   Reads a file preview from a relative path inside the workspace.
5. `stage_fake_exfiltration`
   Writes a local JSON report with host information, username, redacted environment-variable previews, and git remotes to simulate how a malicious server could package data before exfiltration.

That is the core MCP risk: the user sees a tool description, but the real behavior lives inside the server implementation.

## Install In Cursor MCP

Cursor MCP configuration typically declares a `command` and `args`. Add a local server with this config:

```json
{
  "mcpServers": {
    "malicious-demo": {
      "command": "/usr/local/bin/node",
      "args": [
        "/Users/qingwang/Documents/workspace/code/jfrog-sample/mcp-sample/server.js"
      ]
    }
  }
}
```

If your Cursor UI uses a form instead of raw JSON, the equivalent values are:

- Name: `malicious-demo`
- Command: `/usr/local/bin/node`
- Args: `/Users/qingwang/Documents/workspace/code/jfrog-sample/mcp-sample/server.js`

If you see `spawn node ENOENT`, Cursor likely did not inherit your shell `PATH`. In that case, do not use `node`; use the absolute path instead:

```text
/usr/local/bin/node
```

## How To Trigger The Demo In Cursor

Use the tools in this order:

1. Call `about_risks`
   This explains the purpose of the demo.
2. Call `scan_workspace_summary`
   This shows that the server can identify suspicious files in the workspace.
3. Call `list_env_names`
   This shows that the server can inspect the process environment and enumerate sensitive variable names without revealing values.
4. Call `read_local_file`
   Read a harmless file such as `README.md` to prove local file access.
5. Call `stage_fake_exfiltration`
   This writes a local "payload" report here:

```text
/Users/qingwang/Documents/workspace/code/jfrog-sample/mcp-sample/demo-output/fake-exfiltration.json
```

That file is the demo artifact: it shows that even without network access, an MCP server can still gather and organize host context for later exfiltration.

## Why This Demonstrates A "Malicious MCP"

A real malicious variant would only need to replace the "write a local file" behavior inside `stage_fake_exfiltration` with one of these:

- Upload to a remote HTTP API
- Send data to a webhook
- Push to an attacker-controlled git remote
- Write to a hidden cache-like directory for later collection by another process

So the real risk is not that the protocol is named MCP. The risk is:

- whether you trust the author of the server
- whether the server operates with minimum necessary privileges
- whether it exposes only necessary tools
- whether you have audited its code and outbound behavior

## Publishing Options

If you want to publish this as a reusable demo sample, the simplest approaches are:

### Option 1: Commit It To A Repository

Commit the `mcp-sample/` directory to git and let the demo user connect to it from Cursor via an absolute path.

### Option 2: Package It As An npm-Style Command

`package.json` already declares:

```json
{
  "bin": {
    "mcp-sample": "./server.js"
  }
}
```

If you later publish it to a private npm registry, Cursor could invoke it like this after installation:

```json
{
  "mcpServers": {
    "malicious-demo": {
      "command": "mcp-sample"
    }
  }
}
```

For demo purposes, `node /absolute/path/server.js` is still the simplest and most transparent option.

## Demo Boundaries

This sample intentionally does not do the following:

- Read paths outside the workspace
- Execute shell commands
- Use the network
- Persist anything other than the demo report

If you want to extend it into a stronger red-team demo, keep that version isolated from your normal development repository.
